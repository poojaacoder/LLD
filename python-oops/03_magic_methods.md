# 03 ŌĆö Magic / Dunder Methods

## What is this?

Dunder methods (short for "double underscore") are special methods like `__init__`, `__len__`, and `__add__` that Python calls automatically in response to certain operations. They let your custom objects behave exactly like Python's built-in types.

---

## Why Do Dunder Methods Exist?

> Imagine if every library you used had its own way to get the length of a thing ŌĆö `mylist.size()`, `mystring.length()`, `myqueue.count()`. You would have to memorize a different name for every type. Instead, Python says: "every type that wants to support `len()` just needs to define `__len__`." One name, works everywhere.

Dunder methods are Python's protocol system. When you write `len(x)`, Python calls `x.__len__()`. When you write `a + b`, Python calls `a.__add__(b)`. When you write `with x as y:`, Python calls `x.__enter__()`.

By implementing the right dunder methods, your class can:
- Be used with `len()`, `in`, `for`, `bool()`, `abs()`
- Support `==`, `<`, `>` comparisons
- Be used as a key in a `set` or `dict`
- Work with `with` statements
- Be called like a function
- Support `+`, `-`, `*` operators

You have already seen `__init__`, `__repr__`, and `__str__` in file 01. This file covers the rest.

## `__repr__` and `__str__` (Quick Review)

These two are the most important dunder methods after `__init__`. Define them in every class you write.

```python
class Point:
    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y

    def __repr__(self) -> str:
        # For developers ŌĆö unambiguous, use in logs and debugging
        return f"Point({self.x}, {self.y})"

    def __str__(self) -> str:
        # For end users ŌĆö friendly and readable
        return f"({self.x}, {self.y})"

p = Point(3, 4)
print(p)        # (3, 4)       ŌåÉ __str__
print(repr(p))  # Point(3, 4)  ŌåÉ __repr__
```

## Comparison and Hashing

### `__eq__` and `__lt__`

By default, Python uses the object's memory address for `==` comparisons. Two separate objects with identical data are not equal:

```python
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

p1 = Point(1, 2)
p2 = Point(1, 2)
print(p1 == p2)   # False ŌĆö different objects in memory!
```

To make `==` check actual data, define `__eq__`. To make `<` work (for sorting), define `__lt__`.

### `@total_ordering` ŌĆö the shortcut

Defining all six comparison methods (`__eq__`, `__lt__`, `__le__`, `__gt__`, `__ge__`, `__ne__`) is tedious. The `@total_ordering` decorator generates the missing ones for you automatically ŌĆö you only need to write `__eq__` plus one other.

The following example creates `Employee` objects that can be compared by salary and sorted in a list:

```python
from functools import total_ordering

@total_ordering   # generates __gt__, __le__, __ge__, __ne__ automatically
class Employee:
    def __init__(self, name: str, salary: float):
        self.name = name
        self.salary = salary

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Employee):
            return NotImplemented   # let Python try the other object's __eq__
        return self.salary == other.salary

    def __lt__(self, other: "Employee") -> bool:
        return self.salary < other.salary

    def __repr__(self) -> str:
        return f"Employee({self.name!r}, {self.salary})"


employees = [Employee("Bob", 90000), Employee("Alice", 70000), Employee("Charlie", 80000)]
print(sorted(employees))
# [Employee('Alice', 70000), Employee('Charlie', 80000), Employee('Bob', 90000)]

print(Employee("Bob", 90000) > Employee("Alice", 70000))   # True  ŌåÉ generated by @total_ordering
```

> **What just happened?**
> We only wrote `__eq__` and `__lt__`, but `@total_ordering` created `__gt__`, `__le__`, `__ge__`, and `__ne__` for us. Python's `sorted()` uses `__lt__` under the hood, so the list sorts correctly.

### `__hash__`

There is an important rule in Python: **if you define `__eq__`, you must also define `__hash__`**. Here is why:

Python uses `hash()` to put objects into sets and as dictionary keys. By default, an object's hash is based on its memory address. But if you change what `==` means, the hash might no longer be consistent ŌĆö two objects that are "equal" could end up in different hash buckets.

Python's solution: if you define `__eq__`, Python automatically sets `__hash__ = None`, which means your objects can no longer be used in sets or as dict keys. You must opt back in by defining `__hash__`.

```python
from functools import total_ordering

@total_ordering
class Employee:
    def __init__(self, name: str, salary: float):
        self.name = name
        self.salary = salary

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Employee):
            return NotImplemented
        return self.salary == other.salary

    def __lt__(self, other: "Employee") -> bool:
        return self.salary < other.salary

    def __hash__(self) -> int:
        # Use a stable, immutable attribute ŌĆö name is a reasonable choice here
        return hash(self.name)

    def __repr__(self) -> str:
        return f"Employee({self.name!r}, {self.salary})"


# Now Employee objects can be put in sets and used as dict keys
emp_set = {Employee("Alice", 70000), Employee("Bob", 90000)}
print(len(emp_set))   # 2
```

> **Key takeaway:** The rule is: objects that compare equal must have the same hash. If two employees are "equal" because they have the same salary, but you hash by name (which might differ), be aware of the inconsistency. For this reason, the safest approach is to hash only immutable, identity-defining attributes.

## Container Protocol

The container protocol lets your class behave like a Python list, set, or collection. Implement these methods and your class will work with `len()`, `in`, `for`, and `if obj:`.

This `Stack` class wraps a list and exposes a stack interface:

```python
class Stack:
    def __init__(self):
        self._items: list = []

    def push(self, item) -> None:
        self._items.append(item)

    def pop(self):
        return self._items.pop()

    def __len__(self) -> int:
        # Enables: len(stack)
        return len(self._items)

    def __contains__(self, item) -> bool:
        # Enables: item in stack
        return item in self._items

    def __iter__(self):
        # Enables: for x in stack (top to bottom)
        return iter(reversed(self._items))

    def __bool__(self) -> bool:
        # Enables: if stack:  /  bool(stack)
        return len(self._items) > 0

    def __repr__(self) -> str:
        return f"Stack({self._items})"


s = Stack()
s.push(1)
s.push(2)
s.push(3)

print(len(s))        # 3
print(2 in s)        # True
print(list(s))       # [3, 2, 1]  ŌåÉ top to bottom (reversed)
print(bool(s))       # True

s2 = Stack()
if not s2:           # uses __bool__
    print("Empty stack!")
```

> **What just happened?**
> Python translates `len(s)` into `s.__len__()`, `2 in s` into `s.__contains__(2)`, and `for x in s` into `iter(s)` which calls `s.__iter__()`. We never have to call these methods ourselves ŌĆö Python does it automatically.

> **Key takeaway:** If you define `__len__`, Python will use it for `__bool__` automatically (empty = falsy) ŌĆö but it is clearer to define `__bool__` explicitly when you have a specific meaning in mind.

## Context Manager Protocol: `__enter__` and `__exit__`

The context manager protocol powers the `with` statement. You have used it with files: `with open("file.txt") as f:`. It guarantees that setup happens before the block and cleanup happens after ŌĆö even if an exception is raised.

> Think of it like a security guard who opens the door when you arrive and locks it behind you when you leave, no matter what happened inside.

Define `__enter__` (the setup) and `__exit__` (the cleanup) to make your own classes work with `with`.

```python
class DatabaseConnection:
    def __init__(self, url: str):
        self.url = url
        self.connection = None

    def __enter__(self):
        # Called at the start of the with block
        print(f"Connecting to {self.url}")
        self.connection = {"connected": True}   # simulate a connection
        return self.connection                  # this becomes the 'as' variable

    def __exit__(self, exc_type, exc_val, exc_tb) -> bool:
        # Called at the end of the with block (always, even on exception)
        print("Closing connection")
        self.connection = None
        if exc_type is ValueError:
            print(f"Suppressing ValueError: {exc_val}")
            return True    # returning True suppresses the exception
        return False       # returning False (or None) re-raises other exceptions


with DatabaseConnection("postgres://localhost/db") as conn:
    print(conn)    # {'connected': True}
# Closing connection

# Watch what happens with an exception
with DatabaseConnection("postgres://localhost/db") as conn:
    raise ValueError("bad query")
# Connecting to postgres://localhost/db
# Closing connection
# Suppressing ValueError: bad query
# (no crash!)
```

> **What just happened?**
> Python called `__enter__` at the start, and `__exit__` at the end regardless of whether an exception occurred. `__exit__` receives three arguments: the exception type, the exception value, and the traceback. Returning `True` tells Python to suppress the exception; returning `False` lets it propagate.

> **Key takeaway:** Context managers are the right tool whenever you have a paired setup/teardown operation: opening/closing files, acquiring/releasing locks, starting/committing transactions, connecting/disconnecting. They prevent resource leaks.

## Callable Objects: `__call__`

Normally, you call functions with `()`. But if you define `__call__` on a class, instances of that class can also be called with `()`.

> Think of a function that "remembers" some state between calls. A plain function can't do that easily, but a callable object can.

```python
class Multiplier:
    def __init__(self, factor: float):
        self.factor = factor

    def __call__(self, value: float) -> float:
        # Called when you do: multiplier_instance(value)
        return value * self.factor


double = Multiplier(2)
triple = Multiplier(3)

print(double(5))          # 10
print(triple(5))          # 15
print(callable(double))   # True  ŌåÉ callable() checks for __call__

# You can even pass it as a callback
results = list(map(double, [1, 2, 3, 4, 5]))
print(results)   # [2, 4, 6, 8, 10]
```

> **What just happened?**
> `double(5)` looks like a function call, but `double` is an object. Python translates `double(5)` into `double.__call__(5)`. Because `double` carries its `factor` state, it behaves like a "factory function" that has been pre-configured.

> **Key takeaway:** Callable objects are useful for implementing strategies, callbacks, and decorators. In LLD, you might use them for pluggable pricing functions or validators.

## Arithmetic Operators

You can make your objects support `+`, `-`, `*`, and other operators by implementing the arithmetic dunder methods. This is how NumPy arrays support `array1 + array2`.

The `Vector` class below supports addition, subtraction, multiplication by a scalar, and `abs()`:

```python
class Vector:
    def __init__(self, x: float, y: float):
        self.x, self.y = x, y

    def __add__(self, other: "Vector") -> "Vector":
        # v1 + v2
        return Vector(self.x + other.x, self.y + other.y)

    def __sub__(self, other: "Vector") -> "Vector":
        # v1 - v2
        return Vector(self.x - other.x, self.y - other.y)

    def __mul__(self, scalar: float) -> "Vector":
        # v1 * 3  ŌåÉ scalar on the RIGHT
        return Vector(self.x * scalar, self.y * scalar)

    def __rmul__(self, scalar: float) -> "Vector":
        # 3 * v1  ŌåÉ scalar on the LEFT
        # Python tries scalar.__mul__(v1) first; when that fails, tries v1.__rmul__(scalar)
        return self.__mul__(scalar)

    def __abs__(self) -> float:
        # abs(v)  ŌåÉ magnitude of the vector
        return (self.x**2 + self.y**2) ** 0.5

    def __repr__(self) -> str:
        return f"Vector({self.x}, {self.y})"


v1 = Vector(1, 2)
v2 = Vector(3, 4)

print(v1 + v2)    # Vector(4, 6)
print(v2 - v1)    # Vector(2, 2)
print(v1 * 3)     # Vector(3, 6)   ŌåÉ uses __mul__
print(3 * v1)     # Vector(3, 6)   ŌåÉ uses __rmul__
print(abs(v2))    # 5.0
```

> **What just happened?**
> `3 * v1` is interesting. Python first tries `int.__mul__(3, v1)`. When that returns `NotImplemented` (because `int` doesn't know about `Vector`), Python tries `v1.__rmul__(3)`. The "r" in `__rmul__` stands for "reflected" ŌĆö it handles the case where your object is on the right side of the operator.

## Common Mistakes

**1. Defining `__eq__` but forgetting `__hash__`**
```python
class Item:
    def __init__(self, id):
        self.id = id
    def __eq__(self, other):
        return self.id == other.id
    # No __hash__ defined!

item = Item(1)
item_set = {item}   # TypeError: unhashable type: 'Item'
```

**2. Returning `None` from `__add__` instead of a new object**
```python
class Vector:
    def __add__(self, other):
        self.x += other.x   # Wrong: mutates self instead of creating new object
        # Also wrong: doesn't return anything, so v1 + v2 evaluates to None
```

**3. Using `__exit__` wrong ŌĆö returning `True` accidentally**
```python
def __exit__(self, exc_type, exc_val, exc_tb):
    self.close()
    # If you don't explicitly return, Python gets None, which is falsy ŌĆö good
    # But if you write:
    return True   # This suppresses ALL exceptions, including unexpected ones
```

**4. Forgetting `isinstance` check in `__eq__`**
```python
class Employee:
    def __eq__(self, other):
        return self.salary == other.salary   # crashes if other doesn't have .salary

# Right:
def __eq__(self, other):
    if not isinstance(other, Employee):
        return NotImplemented   # let Python try the other side
    return self.salary == other.salary
```

**5. Not returning `self` from in-place operators**
```python
class Vector:
    def __iadd__(self, other):
        self.x += other.x
        self.y += other.y
        # Forgot: return self
        # v += other_v will set v to None!

# Right:
def __iadd__(self, other):
    self.x += other.x
    self.y += other.y
    return self   # must return self for += to work correctly
```

## Quick Reference Table

| Dunder method | Triggered by | What it does |
|---|---|---|
| `__init__(self, ...)` | `MyClass(...)` | Constructor ŌĆö sets up the object |
| `__repr__(self)` | `repr(obj)`, REPL, logs | Developer-friendly string representation |
| `__str__(self)` | `print(obj)`, `str(obj)` | User-friendly string representation |
| `__eq__(self, other)` | `obj == other` | Equality check |
| `__ne__(self, other)` | `obj != other` | Inequality (auto-generated by `@total_ordering`) |
| `__lt__(self, other)` | `obj < other`, `sorted()` | Less-than comparison |
| `__le__(self, other)` | `obj <= other` | Less-than-or-equal |
| `__gt__(self, other)` | `obj > other` | Greater-than |
| `__ge__(self, other)` | `obj >= other` | Greater-than-or-equal |
| `__hash__(self)` | `hash(obj)`, `set`, `dict` key | Hash value ŌĆö must be consistent with `__eq__` |
| `__len__(self)` | `len(obj)` | Number of items |
| `__contains__(self, item)` | `item in obj` | Membership test |
| `__iter__(self)` | `for x in obj`, `list(obj)` | Returns an iterator |
| `__next__(self)` | `next(obj)` | Returns next item from iterator |
| `__bool__(self)` | `bool(obj)`, `if obj:` | Truthiness |
| `__getitem__(self, key)` | `obj[key]` | Index/key access |
| `__setitem__(self, key, val)` | `obj[key] = val` | Index/key assignment |
| `__delitem__(self, key)` | `del obj[key]` | Index/key deletion |
| `__enter__(self)` | `with obj as x:` (start) | Context manager setup |
| `__exit__(self, ...)` | `with obj as x:` (end) | Context manager cleanup |
| `__call__(self, ...)` | `obj(...)` | Makes the object callable like a function |
| `__add__(self, other)` | `obj + other` | Addition |
| `__radd__(self, other)` | `other + obj` | Reflected addition (other side) |
| `__iadd__(self, other)` | `obj += other` | In-place addition |
| `__sub__(self, other)` | `obj - other` | Subtraction |
| `__mul__(self, other)` | `obj * other` | Multiplication |
| `__rmul__(self, other)` | `other * obj` | Reflected multiplication |
| `__truediv__(self, other)` | `obj / other` | Division |
| `__abs__(self)` | `abs(obj)` | Absolute value / magnitude |

## Quick Summary

- Dunder methods are how you make custom classes play nicely with Python's built-in syntax.
- `__repr__` and `__str__` ŌĆö always define them. `__repr__` for developers, `__str__` for end users.
- `__eq__` and `__hash__` ŌĆö define both together. Objects that compare equal must hash equal.
- Use `@total_ordering` to avoid writing six comparison methods by hand.
- The container protocol (`__len__`, `__contains__`, `__iter__`, `__bool__`) makes your class work like a built-in collection.
- The context manager protocol (`__enter__`, `__exit__`) powers the `with` statement ŌĆö use it for any paired setup/cleanup.
- `__call__` lets an object be used like a function ŌĆö great for strategies and callbacks.
- `__rmul__` (and other `__r*__` methods) handle cases where your object is on the right side of an operator.
